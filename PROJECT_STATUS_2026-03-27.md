# Project Status Audit — 2026-03-27

## Краткий вывод

Проект работает как локальный Windows-скрипт для пакетной обработки PDF-счетов с записью в Excel и уведомлениями в Telegram. Основная обработка сейчас построена на `pdfplumber`, наборе supplier-specific extractor-функций и перемещении файлов между папками `EingangsRG`, `manual` и годовыми папками `RG ... RepRG`.

Репозиторий GitHub существует: `https://github.com/mailstoaleksei-droid/repair-invoice-bot`, но текущая локальная папка `PDF_Processor` не является git-репозиторием и не синхронизирована с ним напрямую.

## Что сейчас уже работает

- Есть основной обработчик PDF: `process_pdf_v7_3.py`
- Есть извлечение текста из PDF через `pdfplumber`
- Есть классификация поставщиков и отдельные extractor-функции примерно для 16 поставщиков
- Есть запись результатов в Excel через `openpyxl`
- Есть проверка дубликатов по Excel
- Есть раскладка файлов:
  - успешные в `checked_*`
  - ошибки и нераспознанные в `manual_*` или `error_*`
- Есть журналирование в `logs/processing.log`
- Есть Telegram-модуль уведомлений `unified_telegram.py`
- Есть Telegram-бот `telegram_bot_v4_updated.py`, который запускает обработку и показывает статус

## Что подтверждено по коду

- `process_pdf_v7_3.py` использует локальные папки и Excel-файл через hardcoded paths
- В `process_pdf_v7_3.py` включены Telegram-уведомления и детальные сообщения
- `process_all_pdfs()` обрабатывает входящую папку, пишет в Excel, двигает файлы и возвращает сводку
- `unified_telegram.py` централизует форматирование и throttling уведомлений
- В `requirements.txt` есть зависимости для PDF, Excel, Telegram и watchdog

## Что не совпадает с текущим кодом или требует уточнения

- Локальный `CHECKLIST.md` описывает целевую AI/OpenAI/PostgreSQL-архитектуру, а не текущую реализацию
- OCR для сканов в рабочем коде не реализован: unreadable/empty PDF отправляются в `manual`
- Автоматический мониторинг не готов локально:
  - `telegram_bot_v4_updated.py` импортирует `file_monitor.py`
  - в рабочей папке этого файла нет, он есть только в `_archive`
- Бот ссылается на `process_pdf_v7.py`, а в папке сейчас лежит `process_pdf_v7_3.py`
- Обработка завязана на `Repair_2025.xlsx` и папку `RG 2025 ...`, что выглядит как устаревшая конфигурация для 2026 года
- В коде присутствуют hardcoded Telegram token/chat id, это риск безопасности

## Связь с GitHub

- Удаленный репозиторий существует и доступен: `mailstoaleksei-droid/repair-invoice-bot`
- Локальная папка `PDF_Processor` не содержит `.git`
- Значит:
  - связь с GitHub у проекта как у удаленного репозитория есть
  - связь текущей папки с GitHub локально отсутствует

## Связь с Telegram-ботом

- Связь по коду есть
- `process_pdf_v7_3.py` умеет отправлять детальные уведомления в Telegram
- `telegram_bot_v4_updated.py` умеет запускать обработку через subprocess и показывать статус
- Но интеграция сейчас частично хрупкая:
  - путь к исполняемому скрипту указывает на другую версию файла
  - модуль мониторинга локально отсутствует

## Основные проблемы проекта

- Часто извлекается BRUTTO вместо NETTO
- Некорректно определяется поставщик
- Некорректно извлекаются поля из счетов с новыми layout
- Сканированные PDF не обрабатываются автоматически
- Слишком много логики зашито в один файл
- Нет тестов
- Нет безопасной конфигурации через `.env`
- Нет локальной git-связи с GitHub

## Приоритетный вывод

Если цель проекта — 90% автоматизации, то первый реальный приоритет не AI-рефакторинг, а стабилизация текущего контура:

1. Починить конфигурацию путей и версий файлов
2. Починить Telegram-бот, чтобы он запускал актуальный обработчик
3. Добавить OCR fallback для сканов
4. Вынести supplier rules и конфигурацию из монолитного скрипта
5. Добавить тесты на поставщиков и NETTO/BRUTTO-правила
